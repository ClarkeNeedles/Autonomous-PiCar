## Overview
This session focused on developing the VPFS interface node and integrating it with a temporary odometry-based pose estimation system. The goal was to maintain a continuous estimate of the robot’s position between VPFS updates, while also evaluating whether this additional complexity was necessary.

---

## Agenda
- Implement VPFS interface node  
- Integrate VPFS position data into ROS2 topics  
- Develop odometry-based pose estimation between updates  
- Evaluate necessity of odometry fusion  

---

## Discussion Summary

### VPFS Interface Node
- Implemented the `vpfs_interface_node` to communicate with the VPFS server  
- Responsibilities:
  - Poll robot position using `/whereami/<team_id>`  
  - Publish pose to `/vpfs/pose`  
  - Poll available fares and determine best fare  
  - Handle claim and drop requests via ROS topics  

- The node continuously updates:
  - Robot position (x, y, heading)  
  - Available fares and best fare selection  
  - Current fare state (idle, active, failed states)  

- Position data is published as a `PoseStamped` message for downstream nodes

### Odometry Fusion Approach (Initial Design)
- Implemented an `odometry_fusion_node` to estimate robot pose between VPFS updates  
- Approach:
  - Use `/cmd_vel` to dead-reckon position  
  - Blend this estimate with VPFS pose updates  
  - Maintain smooth, continuous pose estimate  

- Key idea:
  - VPFS provides **discrete updates (low frequency)**  
  - Odometry fills in the gaps using motion integration  

- Features:
  - Dead reckoning using linear and angular velocity  
  - Optional blending parameter (`alpha`) to combine VPFS and odometry  
  - Continuous publishing to `/odometry/filtered`  

- **File Reference:**
  ```

  ```

### Key Insight: VPFS Update Frequency
- After testing, it was observed that:
  - VPFS updates occur **frequently enough (≈1–5 Hz)**  
  - Position estimates are already sufficiently smooth for navigation  

- As a result:
  - The added complexity of odometry fusion is **not necessary**  
  - Dead reckoning introduces potential drift and inconsistency  

---

### Design Pivot
- Decision made to **simplify the system architecture**:
  - Remove reliance on odometry fusion for position tracking  
  - Use **VPFS pose directly as the ground truth position**  

- Benefits:
  - Reduced system complexity  
  - Fewer sources of error (no drift from dead reckoning)  
  - Easier debugging and integration with planner  

- Updated approach:
  - `/vpfs/pose` becomes the primary position source  
  - Downstream nodes (e.g., behavior manager, planner) consume VPFS data directly  

---

## Key Decisions
- VPFS is reliable and frequent enough to act as the sole localization source  
- Remove unnecessary odometry fusion layer  
- Prioritize simplicity and robustness over theoretical completeness
