### What is VPFS?

**VPFS** stands for **Virtual Passenger Fare System**. It is a taxi-dispatch-style coordination layer for the robot. A fare is a pickup-and-dropoff job: the system receives a fare (with a source and destination location), drives the robot to the pickup point, waits for confirmation that the passenger has boarded, then drives to the dropoff point and waits for completion. The system manages one active fare at a time.

### The Three Nodes

The full system is launched by `vpfs_queue_system.sh`, which builds and starts three ROS 2 nodes in parallel:

| Node | Package | Role |
|------|---------|------|
| `vpfs_queue_bridge` | `localization_pkg` | Receives fares from the external queue and publishes them as JSON on `/vpfs/current_fare` |
| `behavior_manager_node` | `behavior_pkg` | Orchestrator — reads fares, tracks robot progress, controls driving enable/disable, drives navigation targets |
| `map_planner` | `navigation_pkg` | Pathfinder — receives current and target poses, runs Dijkstra on a road graph, publishes the next node and turn sequence |

### Node 1: `vpfs_queue_bridge` (localization_pkg)

This node bridges the external VPFS fare queue to the ROS 2 system. It publishes the current active fare as a JSON string on `/vpfs/current_fare`. The JSON payload format consumed by `behavior_manager` is:

```json
{
  "status": "active",
  "id": 3,
  "src":  { "x": 120.0, "y": 85.0 },
  "dest": { "x": 430.0, "y": 210.0 },
  "inPosition": false,
  "pickedUp": false,
  "completed": false
}
```

When there is no active fare, `status` is anything other than `"active"` and the behavior manager clears its navigation state.

### Node 2: `behavior_manager_node` (behavior_pkg)

This is the main orchestrator. It runs a **20 Hz timer loop** (`50 ms`) and manages:

#### Subscriptions

| Topic | Type | Purpose |
|-------|------|---------|
| `/vpfs/current_fare` | `String` (JSON) | Receives fare assignments from the bridge |
| `/odometry/filtered` | `Odometry` | Robot's current position and heading |
| `/ultrasonic/detected` | `Bool` | Obstacle detection — stops driving |

#### Publications

| Topic | Type | Purpose |
|-------|------|---------|
| `/navigation/current_pose` | `Pose2D` | Forwards robot pose to the map planner |
| `/navigation/target_pose` | `Pose2D` | Tells the map planner where to go (pickup or dropoff) |
| `/autonomy/driving_allowed` | `Bool` | Master enable/disable for motor control |

#### Fare Lifecycle State Machine

The behavior manager tracks a `FarePhase` enum with 7 states:

```
NO_FARE
  │
  │  New fare received on /vpfs/current_fare
  ▼
GO_TO_PICKUP  ──────────────────────────────────────────────────┐
  │                                                              │
  │  planarDistance(current, pickup) ≤ 25 cm                    │
  ▼                                                              │
WAIT_FOR_PICKUP_POSITION                                         │
  │                                                              │
  │  fare.inPosition == true                                     │
  ▼                                                              │ WAIT timeout
WAIT_FOR_PICKUP_CONFIRM                                          │ (45 seconds)
  │                                                              │ in any WAIT
  │  fare.pickedUp == true                                       │ state →
  ▼                                                              │ clearFareState()
GO_TO_DROPOFF ◄─────────────────────────────────────────────────┘
  │
  │  planarDistance(current, dropoff) ≤ 25 cm
  ▼
WAIT_FOR_DROPOFF_POSITION
  │
  │  fare.inPosition == true
  ▼
WAIT_FOR_DROPOFF_CONFIRM
  │
  │  fare.completed == true
  ▼
NO_FARE  (clearFareState)
```

#### Driving Enable Logic

`/autonomy/driving_allowed` is published `true` only when all three conditions hold:

```
driving = NOT obstacle_active
          AND NOT obstacle_hold_active
          AND NOT in_wait_state
```

A `WAIT` state (`WAIT_FOR_PICKUP_POSITION`, `WAIT_FOR_PICKUP_CONFIRM`, `WAIT_FOR_DROPOFF_POSITION`, `WAIT_FOR_DROPOFF_CONFIRM`) stops the robot while it waits for external confirmation.

#### Obstacle / Ultrasonic Hold

When `/ultrasonic/detected` goes `true`, the behavior manager starts a **hold timer** of `ultrasonic_wait_sec` (default: **3 seconds**). During the hold, `driving_allowed` is `false`. After the hold expires:
- If obstacle is still present → extend the hold by another 3 seconds
- If obstacle is gone → release the hold and resume

Turn signals are deliberately not touched here — that is handled by the `blink_timer_callback` in the drive node.

### Node 3: `map_planner` (navigation_pkg)

This node implements **Dijkstra's algorithm** on a road graph loaded from two CSV files:

- `map_nodes.csv` — columns: `id, x, y, label`
- `map_edges.csv` — columns: `from, to, cost` (cost defaults to Euclidean distance if omitted)

Coordinates are scaled by `csv_unit_to_cm` (default `1.0`) on load.

#### Subscriptions

| Topic | Type | Purpose |
|-------|------|---------|
| `/navigation/current_pose` | `Pose2D` | Robot's current x, y, heading |
| `/navigation/target_pose` | `Pose2D` | Destination x, y |

#### Publications

| Topic | Type | Purpose |
|-------|------|---------|
| `/navigation/next_node` | `String` | ID of the immediate next graph node to head toward |
| `/navigation/turn_queue` | `Int32MultiArray` | Ordered list of turns at each node: `0`=LEFT, `1`=STRAIGHT, `2`=RIGHT |
| `/navigation/path_nodes` | `String` | Full planned path, e.g. `"A -> B -> C -> D"` |

#### Planning Logic (`try_plan`)

Every time a new current or target pose arrives, `try_plan()` runs:

1. **Find goal node** — nearest graph node to the target pose (`nearest_node`)
2. **Find start edge** — `choose_start_edge` scores every road segment to find the one the robot is currently on (see score formula below)
3. **Run Dijkstra** from the `to` end of the start edge to the goal node
4. **Build turn queue** — classify each intermediate node as LEFT / STRAIGHT / RIGHT
5. **Publish** next node, turn queue, and full path

If no aligned edge is found (robot is too far from any road or heading doesn't match), it falls back to nearest-node planning from the nearest graph node.

| Parameter | Default | Meaning |

## Challenges & Solutions

### Ideation — Alternatives Considered

- **Pure nearest-node start vs. edge-snapping start:** We could always snap to the nearest graph node to begin routing. However, if the robot is mid-edge (between two nodes), this would produce a suboptimal plan that routes backward through the previous node. The `choose_start_edge` approach scores the robot's position along each edge and picks the one it is most likely currently travelling, resulting in more accurate remaining distance estimates.

### Why We Chose Our Final Solution

Dijkstra was chosen over A* because the road graph is small enough (competition track) that exhaustive search is fast, and we did not have reliable heuristic distances available at planning time. The edge-snap start with score-weighted heading and distance bias was chosen to handle the common real-world case of the robot being mid-segment when a new target is assigned.

## Documentation

### Calculations

**1 — Edge Snap Score**

When choosing which road segment the robot is on, every candidate edge is scored. The best (lowest) score wins:

$$
S = C_{\text{remaining edge}} + C_{\text{remaining path}} + 2 \cdot d_{\perp} + 5 \cdot \epsilon_{\theta}
$$

Where:
- $C_{\text{remaining edge}}$ = `edge.cost × (1 − projection)` — cost left on the current edge
- $C_{\text{remaining path}}$ = Dijkstra cost from edge's `to` node to the goal
- $d_{\perp}$ = perpendicular distance from robot to the edge (metres/cm depending on CSV scale)
- $\epsilon_{\theta}$ = heading error in degrees between robot's heading and edge direction
- The weight `5.0` on heading error means a 12° misalignment penalises as much as 60 cm of lateral offset

**2 — Turn Classification (Cross Product)**

At each intermediate node, the turn direction is determined from the incoming and outgoing edge vectors:

$$
\text{cross} = v_{\text{in},x} \cdot v_{\text{out},y} - v_{\text{in},y} \cdot v_{\text{out},x}
$$
$$
\text{dot} = v_{\text{in},x} \cdot v_{\text{out},x} + v_{\text{in},y} \cdot v_{\text{out},y}
$$
$$
\theta = \text{atan2}(\text{cross},\ \text{dot})
$$
$$
\text{direction} = \begin{cases} \text{STRAIGHT} & |\theta| \leq 25° \\ \text{LEFT} & \theta > 25° \\ \text{RIGHT} & \theta < -25° \end{cases}
$$

**3 — Yaw from Odometry Quaternion**

The behavior manager extracts the robot's 2D heading from the `Odometry` quaternion:

$$
\psi = \text{atan2}\!\left(2(w z + x y),\ 1 - 2(y^2 + z^2)\right)
$$

Where $(x, y, z, w)$ are the quaternion components from `/odometry/filtered`. This isolates the yaw (rotation about the vertical axis) from the full 3D orientation.

**4 — Pickup / Dropoff Proximity Check**

$$
d = \sqrt{(\Delta x)^2 + (\Delta y)^2} \leq 25\ \text{cm}
$$

This Euclidean distance check runs every timer tick (50 ms) in `updateFareProgress()`. When `d ≤ 25 cm`, the robot transitions from a `GO_TO_*` phase to the corresponding `WAIT_FOR_*` phase.

### Code Snippets

**Snippet 1 — Edge Snap Start Selection (`choose_start_edge`)**

For each road segment in the graph, the robot's position is projected onto the segment. Segments that are too far away or pointing the wrong direction are rejected. The remaining candidates are scored and the best is returned:

```cpp
// Project robot pose onto each edge segment
const double projection = ((pose_dx * segment_dx) + (pose_dy * segment_dy))
                          / segment_length_sq;
const double clamped_projection = std::clamp(projection, 0.0, 1.0);

// Perpendicular distance from robot to the closest point on the segment
const double closest_x = from_node.x + (clamped_projection * segment_dx);
const double closest_y = from_node.y + (clamped_projection * segment_dy);
const double distance_to_edge = std::sqrt(
    ((pose.x - closest_x) * (pose.x - closest_x)) +
    ((pose.y - closest_y) * (pose.y - closest_y)));

// Reject if too far or heading is too misaligned
if (distance_to_edge > max_snap_distance) continue;
const double heading_error_deg = std::abs(
    rad_to_deg(normalize_angle_rad(edge_heading - pose.heading_rad)));
if (heading_error_deg > max_heading_error_deg) continue;

// Score = remaining edge cost + remaining path cost
//         + 2 × lateral offset + 5 × heading error
const double score =
    remaining_edge_cost +
    remaining_path.total_cost +
    (distance_to_edge * 2.0) +
    (heading_error_deg * 5.0);
```

**Snippet 2 — Turn Classification (`classify_turn`)**

Cross product of the incoming and outgoing edge vectors gives a signed angle. The sign determines left vs. right; magnitude determines if it's sharp enough to count as a turn:

```cpp
const double cross = (in_x * out_y) - (in_y * out_x);
const double dot   = (in_x * out_x) + (in_y * out_y);
const double angle_deg = rad_to_deg(std::atan2(cross, dot));

if (std::abs(angle_deg) <= straight_threshold_deg) return STRAIGHT;
return angle_deg > 0.0 ? LEFT : RIGHT;
// LEFT=0, STRAIGHT=1, RIGHT=2 — published as Int32MultiArray on /navigation/turn_queue
```

**Snippet 3 — Fare Phase Transitions (`currentFareCallback` + `updateFareProgress`)**

The behavior manager transitions between fare phases based on two independent signals — proximity (from odometry) and VPFS confirmation (from the JSON topic):

```cpp
// Proximity check — runs in onTimer() → updateFareProgress() at 20 Hz
if (fare_phase_ == GO_TO_PICKUP && have_pickup_pose_) {
    const double d = planarDistance(current_pose_, pickup_pose_);
    if (d <= pickup_reached_threshold_cm_) {         // default 25 cm
        fare_phase_ = WAIT_FOR_PICKUP_POSITION;
        fare_wait_entered_ = this->now();
    }
}

// Confirmation check — runs in currentFareCallback() on each JSON update
if (fare_phase_ == WAIT_FOR_PICKUP_POSITION && fare_in_position_) {
    fare_phase_ = WAIT_FOR_PICKUP_CONFIRM;
}
if (fare_phase_ == WAIT_FOR_PICKUP_CONFIRM && fare_picked_up_) {
    fare_phase_ = GO_TO_DROPOFF;
    updatePlannerTargetForFarePhase();   // pushes dropoff to /navigation/target_pose
}

// Timeout — abandons stuck fares after 45 s in any WAIT state
const double elapsed = (this->now() - fare_wait_entered_).seconds();
if (elapsed > fare_wait_timeout_sec_) {
    clearFareState();
}
```
